---
description: Generate proposal.md and spec.md for a new change
---

Generate DEFINE phase artifacts. **Does NOT generate tasks.** Use `/plan` to break specs into tasks, then `/do` to execute.

## Modes

### No arguments: Interactive mode
1. Ask what the user wants to build
2. Explore the codebase for context (invoke `define` for divergent thinking)
3. Propose a change name and generate both artifacts
4. After generation, ask if they want to continue to `/plan`

### /spec <name>: Direct generation
1. Generate proposal: `docs/rein/changes/<name>/proposal.md` — Why, What Changes, Goals, Non-Goals, Assumptions, Open Questions
2. Generate spec: `docs/rein/changes/<name>/spec.md` — Requirements (WHEN/THEN/TEST), Decisions, Risks

### /spec --validate: Validate artifacts
Check that the current spec is complete (has Requirements, Decisions, Risks).

## Content Rules

**proposal.md** must contain:
- **Why** — Problem framing and motivation
- **What Changes** — Direction and scope of change
- **Goals** — What success looks like
- **Non-Goals** — Explicitly out of scope (with reasons)
- **Key Assumptions** — With validation strategies
- **Open Questions** — Unresolved items

**spec.md** must contain:
- **Requirements**: Capabilities with WHEN/THEN scenarios (optionally with `**TEST**` field)
- **Decisions**: Key technical choices and rationale (`- **Decision:** ... — **Rationale:** ...`)
- **Risks / Trade-offs**: Known risks and mitigations

If `proposal.md` already exists in the feature directory, read it for Goals/Non-Goals context before generating spec.md.

spec.md must NOT contain: task lists, implementation ordering, or acceptance criteria (those belong in task.md, generated by /plan).

## Output

After generation, report:
```
Created proposal: docs/rein/changes/<name>/proposal.md
Created spec: docs/rein/changes/<name>/spec.md

Next step: /plan to break this into tasks, then /do to implement
```
