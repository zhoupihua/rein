# Ralph Loop: 结构化重构循环

## 核心原则

**重构 = 在不改变外部行为的前提下改善代码内部结构。**

每一步都必须可回退，每一次改动都必须有测试守护。

## Ralph Loop 流程

```
  ┌──────────────────────────────────────────┐
  │          RALPH LOOP（循环执行）            │
  │                                          │
  │  1. 识别坏味道                            │
  │         │                                │
  │         ▼                                │
  │  2. 编写特征测试（锁定当前行为）            │
  │         │                                │
  │         ▼                                │
  │  3. 小步重构（一次只改一个味道）            │
  │         │                                │
  │         ▼                                │
  │  4. 运行测试（确认行为未变）               │
  │         │                                │
  │         ▼                                │
  │  5. 提交（可回退的增量）                   │
  │         │                                │
  │         └──── 回到步骤1 ────┘             │
  │                                          │
  │  退出条件：无更多坏味道 或 投入产出比不足    │
  └──────────────────────────────────────────┘
```

## 步骤详解

### 步骤1：识别坏味道

扫描目标代码，对照下面的坏味道清单，标记需要改善的点。每次循环只选**一个**最值得修的味道。

优先级：影响理解和修改的味道 > 影响性能的味道 > 纯风格的味道

### 步骤2：编写特征测试（Characterization Test）

重构前**必须**有测试守护当前行为。如果已有充分测试覆盖，跳到步骤3。

特征测试与 TDD 的区别：
- TDD：先写失败测试，再写实现让测试通过
- 特征测试：先观察现有行为，写测试**记录**它（即使行为看起来不对）

```
特征测试编写步骤：
1. 选定要重构的函数/模块
2. 对每个公开输入，记录实际输出（不管是否符合"预期"）
3. 编写测试断言这些输入-输出对
4. 运行测试确认全部通过（证明你正确记录了行为）
```

**关键：** 如果现有行为有 bug，特征测试会记录 bug 的行为。先重构结构，再单独修 bug——一次只做一件事。

### 步骤3：小步重构

选一个最小改动来消除选中的坏味道。遵循以下规则：

- 一次只改一种味道
- 每步改动不超过 ~30 行
- 只改结构，不改行为
- 如果不确定改动是否安全，拆成更小的步

**安全重构手法**（行为不变，可机械执行）：

| 手法 | 适用场景 | 风险 |
|------|---------|------|
| 重命名变量/函数 | 名字不表意 | 极低 |
| 提取函数 | 长函数/重复代码 | 低 |
| 内联函数 | 函数只被调用一次 | 极低 |
| 移动代码 | 代码放错了位置 | 低 |
| 替换临时变量 | 临时变量增加理解负担 | 低 |
| 分解条件表达式 | 复杂 if/else | 低 |
| 合并条件表达式 | 多个条件判断同一件事 | 低 |
| 以多态替代条件 | type-switch 模式 | 中 |
| 引入参数对象 | 参数过多 | 低 |
| 封装集合 | 直接暴露内部集合 | 中 |

### 步骤4：运行测试

每步重构后**立即**运行测试：

```
测试全部通过 → 行为未变，继续步骤5
测试失败 → 立即回退（git checkout 或 undo），重新评估重构手法
```

不要在测试失败的状态下继续重构——错误会叠加。

### 步骤5：提交

每完成一个味道的消除就提交，提交信息格式：

```
refactor: <重构手法> — <消除的坏味道>

例如：
refactor: extract function — 消除长函数 UserHandler.Process
refactor: rename — 变量 d → deadline 更表意
```

小步提交确保任何一步都可独立回退。

## 坏味道清单

### 高优先级（严重影响理解和修改）

| 坏味道 | 特征 | 首选手法 |
|--------|------|---------|
| 长函数 | 函数超过 30 行 | 提取函数 |
| 重复代码 | 同样逻辑出现 2+ 处 | 提取函数 / 上移方法 |
| 深嵌套 | 3+ 层 if/for/while | 卫语句 / 提取函数 |
| 发散式变化 | 一个函数因多种原因修改 | 拆分为多个函数 |
| 霰弹式修改 | 一个变化要改多个函数 | 移动函数 / 内联类 |
| 依恋情节 | 函数大量使用其他类的数据 | 移动函数到数据所在类 |
| 过长参数列表 | 参数超过 4 个 | 引入参数对象 |

### 中优先级（影响结构和扩展）

| 坏味道 | 特征 | 首选手法 |
|--------|------|---------|
| 中间人 | 类只委托调用，无自身逻辑 | 内联函数 / 移除中间人 |
| 不当亲密 | 两个类过度访问彼此私有数据 | 移动函数 / 提取类 |
| 异曲同工的类 | 两个类做同样的事 | 合并 / 选一个删除 |
| 冗余类 | 类几乎无用 | 内联类 |
| 过度通用 | 只有一个实现的抽象 | 简化 / 移除抽象层 |
| 临时字段 | 字段只在某些状态下有值 | 提取子类 / 引入空对象 |

### 低优先级（风格和整洁度）

| 坏味道 | 特征 | 首选手法 |
|--------|------|---------|
| 魔法数字 | 硬编码常量 | 提取常量 |
| 死代码 | 未被调用的函数/变量 | 删除 |
| 注释过度 | 注释解释"怎么做"而非"为什么" | 重命名让代码自解释 |
| 不一致命名 | 同一概念用不同名字 | 统一命名 |

## 何时使用 Ralph Loop

| 场景 | 触发方式 |
|------|---------|
| TDD 的 REFACTOR 阶段 | 自动进入 Ralph loop |
| code-review 发现架构问题 | 审查后启动 Ralph loop 修复 |
| 单纯重构无新功能 | 直接启动 Ralph loop |
| 技术债务清理 | 按坏味道优先级逐个消除 |

## 与 TDD 的关系

```
TDD Red-Green-Refactor 循环：

  RED → GREEN → REFACTOR
                    │
                    ▼
            ┌── Ralph Loop ──┐
            │ 1. 识别坏味道   │
            │ 2. 特征测试     │  ← GREEN 阶段已提供测试守护
            │ 3. 小步重构     │     可跳过步骤2
            │ 4. 运行测试     │
            │ 5. 提交         │
            │ └── 循环 ──┘    │
            └─────────────────┘
```

在 TDD 流程中，GREEN 阶段已经提供了测试守护，Ralph loop 的步骤2（特征测试）通常可以跳过。Ralph loop 主要增强 REFACTOR 阶段的结构性。

## 退出条件

Ralph loop 不是无限循环。在以下情况退出：

1. **无更多值得修的坏味道** — 剩余味道的改善收益不足以抵消改动风险
2. **投入产出比不足** — 修一个味道需要改动大量代码，风险 > 收益
3. **发现需要新功能** — 重构过程中发现缺少能力，先回到 DEFINE 阶段
4. **时间约束** — 达到时间预算，当前状态已比初始状态好，先停下

退出后做一次完整测试运行，确认所有测试通过。

## 反模式

- 一次改多个坏味道（错误来源无法定位）
- 重构时顺便修 bug（混淆行为变更和结构改善）
- 没有测试就重构（无法确认行为未变）
- 重构中途换方向（半成品比原始代码更糟）
- 重命名的同时改逻辑（两个变更混在一起）
- 跳过提交（无法回退到已知良好状态）

## 常见借口

| 借口 | 现实 |
|------|------|
| "改一点就行，不用写测试" | 一点变两点，两点变三点，最终行为失控 |
| "重构太慢了" | 不重构更慢——每次改动都害怕引入 bug |
| "以后再重构" | 以后 = 永远不会。越晚重构成本越高 |
| "这段代码太烂了重写吧" | 重写风险远大于渐进重构，除非有完整规格 |
| "改了行为也没关系" | 那不是重构，是重写。先明确你在做什么 |

## 验证清单

每次 Ralph loop 退出后确认：

- [ ] 每步重构都有测试守护
- [ ] 每步重构都单独提交
- [ ] 所有测试通过
- [ ] 无行为变更（输入输出对与重构前一致）
- [ ] 代码比初始状态更易理解
- [ ] 无半成品重构（提取了一半的函数等）
